Concepts in SSE Models

Structured document

A structured document (see IStructuredDocument) is a document that is capable of being parsed into semantically meaningful regions (see IStructuredDocumentRegion). These regions are one level higher than tokens, or more precisely text regions (see ITextRegion), but smaller than a complete semantic model unit. For example, the following text
<tagname attribute="value"> some info </tagname>
consist of is 3 structured document regions.
<tagname attribute="value">
some info
</tagname>

When interpreted as XML, it has three semantic units, the Element Node named tagname, the Attribute named attribute and a Text Node of ' some info'.

The first of the IStructuredDocumentRegions, consists of 8 ITextRegions,
<
tagname
attribute
=
"
value
"
>

A document is parsed into IStructuredDocumentRegions (and ITextRegions) by a parser (see RegionParser).

The motivation behind this concept of a document is that the "intermediate level" of document regions form the basis of incremental re-parsing (see IStructuredTextReParser). Typically, only the document regions "touched" by a change need to be reparsed.

When a document is changed, it notifies listeners of the document changed event (see StructuredDocumentEvent) which precisely describes the change in terms of the regions. Typically structured models are one such listner, which knows how to interpret the region types to reparse the semantic representation of the document.

StructuredModel

A structured model's (see IStructuredModel) most salient feature is that is has as core data a structured doucment. This allows it to use the structured document regions in its own parsing algorithms and data structures. Clients typically can either change the structured model directly, in which case the document is updated appropriately, or update the doucment, in which case the model is updated appropriately.

ModelManager

Its typically desirable for a model to be shared between various clients, without them needing to be aware of each other. The ModelManger (see IModelManger) serves this role. Some clients desire to share the model and have responsibility to save the model to a resource, if the model is changed. Others clients desire to chare the model and do not care if the changed model is saved or not. Thus, there's two forms of access to a model (currently called 'edit' and 'read', though not great names since either can, technically modify the model).
