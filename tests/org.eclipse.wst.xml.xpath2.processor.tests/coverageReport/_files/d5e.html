106/**
107 * <p>This class provides a convenient behaviour-only 
108 * extension mechanism for the ASTNode hierarchy.
109 * If you feel like you would like to add a method to
110 * the ASTNode hierarchy (or a subtree of the hierarchy),
111 * and you want to have different implementations
112 * of it at different points in the hierarchy,
113 * simply create a HierarchicalASTVisitor representing
114 * the new method and all its implementations,
115 * locating each implementation within the right
116 * visit(XX) method.  If you wanted to add a method implementation to abstract
117 * class Foo, an ASTNode descendant, put your implementation in visit(Foo). 
118 * This class will provide appropriate dispatch, just as if the method
119 * implementations had been added to the ASTNode hierarchy.
120 * </p>
121 * 
122 * <p><b>Details:<b></p>
123 * 
124 * <p>This class has a visit(XX node) method for every for every 
125 * class (concrete or abstract) XX in the ASTNode hierarchy. In this class'
126 * default implementations of these methods, the method corresponding to a given
127 * ASTNode descendant class will call (and return the return value of) the
128 * visit(YY) method for it's superclass YY, with the exception of the
129 * visit(ASTNode) method which simply returns true, since ASTNode doesn't have a
130 * superclass that is within the ASTNode hierarchy.
131 * </p>
132 * 
133 * <p>Because of this organization, when visit(XX) methods  are overridden in a
134 * subclass, and the visitor is applied to a node, only the most specialized
135 * overridden method implementation for the node's type will be called, unless
136 * this most specialized method calls other visit methods (this is discouraged)
137 * or, (preferably) calls super.visit(XX node), (the reference type of the
138 * parameter must be XX) which will invoke this class' implementation of the
139 * method, which will, in turn, invoke the visit(YY) method corresponding to the
140 * superclass, YY.
141 * </p>
142 * 
143 * <p>Thus, the dispatching behaviour achieved when 
144 * HierarchicalASTVisitors' visit(XX) methods, corresponding to a particular
145 * concrete or abstract ASTNode descendant class, are overridden is exactly
146 * analogous to the dispatching behaviour obtained when method implementations
147 * are added to the same ASTNode descendant classes.
148 * </p>
149 */
150/*

153 * Provisional API: This class/interface is part of an interim API that is still under development and expected to
154 * change significantly before reaching stability. It is being made available at this early stage to solicit feedback
155 * from pioneering adopters on the understanding that any code that uses this API will almost certainly be broken
156 * (repeatedly) as the API evolves.
