23/**
24 * Abstract superclass of all Abstract Syntax Tree (AST) node types.
25 * <p>
26 * An AST node represents a JavaScript source code construct, such
27 * as a name, type, expression, statement, or declaration.
28 * </p>
29 * <p>
30 * Each AST node belongs to a unique AST instance, called the owning AST.
31 * The children of an AST node always have the same owner as their parent node.
32 * If a node from one AST is to be added to a different AST, the subtree must
33 * be cloned first to ensure that the added nodes have the correct owning AST.
34 * </p>
35 * <p>
36 * When an AST node is part of an AST, it has a unique parent node.
37 * Clients can navigate upwards, from child to parent, as well as downwards,
38 * from parent to child. Newly created nodes are unparented. When an
39 * unparented node is set as a child of a node (using a
40 * <code>set<i>CHILD</i></code> method), its parent link is set automatically
41 * and the parent link of the former child is set to <code>null</code>.
42 * For nodes with properties that include a list of children (for example,
43 * <code>Block</code> whose <code>statements</code> property is a list
44 * of statements), adding or removing an element to/for the list property
45 * automatically updates the parent links. These lists support the
46 * <code>List.set</code> method; however, the constraint that the same
47 * node cannot appear more than once means that this method cannot be used
48 * to swap elements without first removing the node.
49 * </p>
50 * <p>
51 * ASTs must not contain cycles. All operations that could create a cycle
52 * detect this possibility and fail.
53 * </p>
54 * <p>
55 * ASTs do not contain "holes" (missing subtrees). If a node is required to
56 * have a certain property, a syntactically plausible initial value is
57 * always supplied.
58 * </p>
59 * <p>
60 * The hierarchy of AST node types has some convenient groupings marked
61 * by abstract superclasses:
62 * <ul>
63 * <li>expressions - <code>Expression</code></li>
64 * <li>names - <code>Name</code> (a sub-kind of expression)</li>
65 * <li>statements - <code>Statement</code></li>
66 * <li>types - <code>Type</code></li>
67 * <li>type body declarations - <code>BodyDeclaration</code></li>
68 * </ul>
69 * </p>
70 * <p>
71 * Abstract syntax trees may be hand constructed by clients, using the
72 * <code>new<i>TYPE</i></code> factory methods (see <code>AST</code>) to
73 * create new nodes, and the various <code>set<i>CHILD</i></code> methods
74 * to connect them together.
75 * </p>
76 * <p>
77 * The class {@link ASTParser} parses a string
78 * containing a JavaScript source code and returns an abstract syntax tree
79 * for it. The resulting nodes carry source ranges relating the node back to
80 * the original source characters. The source range covers the construct
81 * as a whole.
82 * </p>
83 * <p>
84 * Each AST node carries bit flags, which may convey additional information about
85 * the node. For instance, the parser uses a flag to indicate a syntax error.
86 * Newly created nodes have no flags set.
87 * </p>
88 * <p>
89 * Each AST node is capable of carrying an open-ended collection of
90 * client-defined properties. Newly created nodes have none.
91 * <code>getProperty</code> and <code>setProperty</code> are used to access
92 * these properties.
93 * </p>
94 * <p>
95 * AST nodes are thread-safe for readers provided there are no active writers.
96 * If one thread is modifying an AST, including creating new nodes or cloning
97 * existing ones, it is <b>not</b> safe for another thread to read, visit,
98 * write, create, or clone <em>any</em> of the nodes on the same AST.
99 * When synchronization is required, consider using the common AST
100 * object that owns the node; that is, use
101 * <code>synchronize (node.getAST()) {...}</code>.
102 * </p>
103 * <p>
104 * ASTs also support the visitor pattern; see the class <code>ASTVisitor</code>
105 * for details.
106 * </p>
107 * <p>
108 * JavaScript units created by <code>ASTParser</code> from a
109 * source document can be serialized after arbitrary modifications
110 * with minimal loss of original formatting. See
111 * {@link JavaScriptUnit#recordModifications()} for details.
112 * See also {@link org.eclipse.wst.jsdt.core.dom.rewrite.ASTRewrite} for
113 * an alternative way to describe and serialize changes to a
114 * read-only AST.
115 * </p>
116 * This class is not intended to be subclassed by clients.
117 *
118 * @see ASTParser
119 * @see ASTVisitor
120 *
121 * Provisional API: This class/interface is part of an interim API that is still under development and expected to
